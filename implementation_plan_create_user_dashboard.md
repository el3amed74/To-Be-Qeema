# Planning User Creation from Super-Admin Dashboard

This feature allows a super-admin to create new users from the dashboard. Based on the [User](file:///c:/laragon/www/tobe/app/Models/User.php#14-139) model, database migrations, and existing architecture (Spatie roles, etc.), here is the plan for the required and optional fields.

## Field Requirements

### 1. Essential User Data
These fields map directly to the `users` table and are fundamental for the user's identity and authentication.

*   **Name (`name`)**:
    *   **Type:** String (Text input)
    *   **Requirement:** **Required**
    *   **Validation:** `required|string|max:255`
*   **Email (`email`)**:
    *   **Type:** String (Email input)
    *   **Requirement:** **Required** if phone number is not provided.
    *   **Validation:** `nullable|email|unique:users,email|required_without:phone_number`
*   **Phone Number (`phone_number`)**:
    *   **Type:** String (Text/Tel input)
    *   **Requirement:** **Required** if email is not provided.
    *   **Validation:** `nullable|string|max:15|unique:users,phone_number|required_without:email`
*   **Password (`password`)**:
    *   **Requirement:** None from the request. The system will auto-generate the password and set it to `"password"`.

### 2. Authorization (Roles)
Since the system uses `spatie/laravel-permission`, assigning a role upon creation is crucial.

*   **Role (`roles`)**:
    *   **Type:** Array of IDs or Names (Multi-select dropdown)
    *   **Requirement:** **Optional** (A user can exist without a role, perhaps as a normal app user, but from the admin dashboard, assigning a role is usually the goal).
    *   **Data Source:** Fetched from the `roles` table.
    *   **Validation:** `nullable|array`, `roles.*: exists:roles,name` (or `id`).

### 3. Extra Considerations (Nice to Have / System Specific)

*   **FCM Token (`fcm_token`)**: This is usually generated by the mobile app/frontend upon login, not entered during admin creation. It should be ignored in this form.

## Proposed Request Payload (Validation Rules)

To implement this, we will need a `StoreUserRequest`. 

```php
[
    'name' => 'required|string|max:255',
    'email' => 'nullable|email|unique:users,email|required_without:phone_number',
    'phone_number' => 'nullable|string|max:15|unique:users,phone_number|required_without:email',
    // 'password' is not needed in the request as we default it to "password"
    'roles' => 'nullable|array',
    'roles.*' => 'exists:roles,name', // Assumes sending role names
    // Mentor details (only required if 'mentor' is selected in roles)
    'bank_name' => 'required_if:roles.*,mentor|string|max:255',
    'bank_account_name' => 'required_if:roles.*,mentor|string|max:255',
    'bank_account_number' => 'required_if:roles.*,mentor|string|max:255',
    'deduction_type' => 'required_if:roles.*,mentor|in:percentage,fixed',
    'deduction_value' => 'required_if:roles.*,mentor|numeric|min:0',
]
```

## User Experience (Frontend Flow)
Since you are building an API for a frontend developer, here is what the frontend needs to do:
1.  **Form Fields:** Name, Email, Phone Number.
2.  **Role Selection:** A dropdown or multi-select of available roles. *Yes, you must provide an API endpoint for the frontend to fetch the list of available roles (e.g., `GET /api/dashboard/roles`) so they can populate this dropdown dynamically.*
3.  **Conditional Mentor Fields:** If the frontend detects that the user selected the "mentor" role from the dropdown, it must immediately show the extra bank and deduction fields for the admin to fill out.
4.  **Submit:** Send the payload to the API.

## Backend Action
1.  Create [User](file:///c:/laragon/www/tobe/app/Models/User.php#14-139) with the provided Name, Email, Phone.
2.  Set `password` to a hashed version of `"password"` by default.
3.  (Wallet is auto-created via `User::booted`).
4.  Sync roles using `$user->syncRoles($request->roles)`.
5.  If `in_array('mentor', $request->roles)` is true, create a `MentorDetail` record attached to the user using the provided bank/deduction fields.
